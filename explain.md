## 1. Complete Directory Structure Explained

The repository `c:\Users\imdha\OneDrive\Documents\ERP-UI\ERP-UI\` is organized following standard Java package conventions and is broken into distinct modules for maintaining the **Model-View-Controller (MVC)** architectural pattern. 

### Root Folders
- **`src/com/erp/`**: The core Java source code for the entire application.
- **`database_team/`**: Contains raw SQL initialization properties or documentation from the backend team (e.g., `application-rds-template.properties`).
- **`src/main/resources/`**: Where application configuration files like `application-rds.properties` are stored, which `BOMService` uses to connect to the database.
- **`lib/`**: Contains external SDKs and libraries (like `erp-sdk.jar`).
- **`build/` & `out/`**: Compiled `.class` files generated by the Java compiler.

### Inside `src/com/erp/`
The code is split into logical MVC packages:
- **`view/` (The View Layer)**
  - **`panels/`**: Contains the root panels for various subsystems.
    - **`manufacturing/`**: Holds all the Swing UI components (Tabs, Panels, Dialogs) specifically for Manufacturing. These classes *only* draw UI elements and never run SQL directly.
    - **`hr/`, `orders/`, `supplychain/`**: Sibling modules representing other subsystems in the ERP.
  - **`facade/`**: View-layer facades that simplify complex UI aggregations.
- **`service/` (The Controller Layer)**
  - Contains services like `BOMService.java` and `InventoryApiServer.java`. These classes act as the "brain." They take inputs from the UI tabs and translate them into database commands or network calls.
- **`util/` (The Utility Layer)**
  - Contains helper classes like `JSONUtil.java` (for serializing BOM trees) and `UIHelper.java` (for globally defining fonts, colors, and button styles).
- **`exceptions/`**
  - Contains custom business-logic exceptions like `InvalidQuantityInputException` and `QcDefectThresholdExceededException`.

## 2. Modules Explained Like You're 5 Years Old (ELI5)

Imagine you are running a giant toy factory. Here is how each part of the app helps you:

1. **BOM Explorer (Bill of Materials):** This is your **Recipe Book**. If you want to build a toy car, this tab tells you that you need 4 wheels, 1 engine, and 1 plastic body. 
2. **Manufacturing Planning (MRP):** This is your **Calculator**. If a customer orders 100 toy cars, this tab calculates exactly how much plastic you need to buy and how many hours it will take your workers to build them.
3. **Routing & Work Centers:** This is your **Factory Floor Blueprint**. It tells the workers: "First, go to the painting room. Then, go to the assembly room."
4. **Assembly Lines:** This is the **Conveyor Belt**. It physically tracks a batch of toy cars moving from one room to the next until they are finished.
5. **Shop Floor Execution:** This is the **Worker's Notebook**. At the end of the day, a worker writes down, "I finished 50 wheels today."
6. **Quality Control:** This is the **Toy Tester**. Before the toys are boxed up, a tester checks them. If more than 5 out of 100 toys are broken, they stop the whole batch.

## 3. SCM External API & Ngrok Integration

The Manufacturing module needs to talk to the Supply Chain Management (SCM) module, which might be running on a completely different computer.

### How it works:
1. **The Server (`InventoryApiServer.java`):** We use a lightweight, built-in Java `HttpServer` listening on **Port 8081**. 
2. **Ngrok:** Because our computer is behind a firewall, we run a program called **Ngrok**. Ngrok gives us a public URL. Whenever the Supply Chain wants to send us a new material, they send an HTTP POST request to that Ngrok URL. Ngrok securely tunnels that request directly to Port 8081 on our computer.
3. **The Client (`InventoryApiClient.java`):** When *we* create a new material, we need to tell Supply Chain. We use `InventoryApiClient` to send an HTTP POST request to *their* Ngrok URL asynchronously.

## 4. Deep Dive: The BOM Module Code & Patterns

The BOM module handles recursive recipe structures. It leverages the **Composite Pattern** to allow materials to contain sub-materials infinitely deep.

### 4.1 `JSONUtil.java` (The Data Structure)
**Full Path:** `c:\Users\imdha\OneDrive\Documents\ERP-UI\ERP-UI\src\com\erp\util\JSONUtil.java`

*Design Patterns Used:* **Composite Pattern**, **Single Responsibility Principle**.

The `BOMNode` acts as a component that can either be a leaf (raw material) or a composite branch (a sub-assembly that holds a `List<BOMNode> children`).

```java
public class JSONUtil {
    // COMPOSITE PATTERN: A node contains a list of its own type.
    public static class BOMNode {
        public String name;
        public double qty;
        public String uom;
        public double cost;
        public List<BOMNode> children = new ArrayList<>(); // Holds sub-assemblies
        
        public BOMNode(String name, double qty, String uom, double cost) {
            this.name = name; this.qty = qty; this.uom = uom; this.cost = cost;
        }
    }
    
    // SINGLE RESPONSIBILITY: This class ONLY handles JSON parsing.
    public static String toJSON(List<BOMNode> nodes) {
        if (nodes == null) return "[]";
        StringBuilder sb = new StringBuilder();
        sb.append("[");
        for (int i = 0; i < nodes.size(); i++) {
            BOMNode node = nodes.get(i);
            sb.append("{");
            sb.append("\"name\":\"").append(escape(node.name)).append("\",");
            sb.append("\"qty\":").append(node.qty).append(",");
            sb.append("\"cost\":").append(node.cost).append(",");
            sb.append("\"children\":").append(toJSON(node.children)); // Recursive call!
            sb.append("}");
            if (i < nodes.size() - 1) sb.append(",");
        }
        sb.append("]");
        return sb.toString();
    }
}
```

The Assembly Line module dynamically tracks orders as they move through various workstations.

The UI requires a lightweight object to populate drop-downs. It uses an internal DTO `OrderItem`.

    static class OrderItem {
        int id; String name; String status; int currentLine;

    private void assignOrder() {
        OrderItem o = (OrderItem) orderCombo.getSelectedItem();
        int firstLineId = (int) linesTableModel.getValueAt(0, 0);

            BOMService.getInstance().assignOrderToLine(o.id, firstLineId);
            refreshData();
        } catch (Exception e) {
            JOptionPane.showMessageDialog(this, "Error: " + e.getMessage());
        }
    }
}

### 5.2 `BOMService.java` (The Controller / Business Logic)
**Full Path:** `c:\Users\imdha\OneDrive\Documents\ERP-UI\ERP-UI\src\com\erp\service\BOMService.java`

*Design Patterns Used:* **Facade Pattern**, **Singleton Pattern**, **Dependency Inversion Principle**.

`BOMService` is the massive traffic cop of the module. It hides complex business logic (like figuring out which assembly line sequence comes next) from the UI.

```java
public class BOMService {

    // SINGLETON PATTERN: Prevents multiple DB connection pools from opening
    private static BOMService instance;
    private Manufacturing mfg; // DEPENDENCY INVERSION: Depends on Interface, not SQL driver

    private BOMService() {
        // Initializes the abstract SDK
        DatabaseConfig config = DatabaseConfig.fromProperties(Path.of("src", "main", "resources", "application-rds.properties"));
        mfg = (Manufacturing) SubsystemFactory.create(SubsystemName.MANUFACTURING, config);
    }

    public static BOMService getInstance() {
        if (instance == null) instance = new BOMService();
        return instance;
    }

    // FACADE PATTERN: Complex multi-table business logic hidden behind one simple method
    public void moveOrderToNextLine(int orderId, int currentLineId) {
        List<Map<String, Object>> lines = getAllAssemblyLines();
        
        // Step 1: Find the sequence number of the current line
        int currentSeq = -1;
        for (Map<String, Object> l : lines) {
            if (((Number) l.get("line_id")).intValue() == currentLineId) {
                currentSeq = ((Number) l.get("sequence_num")).intValue();
                break;
            }
        }
        
        // Step 2: Find the line with the SMALLEST sequence number GREATER than currentSeq
        int nextLineId = -1;
        int nextSeq = Integer.MAX_VALUE;
        for (Map<String, Object> l : lines) {
            Number lineOrderId = (Number) l.get("production_order_id");
            if (lineOrderId != null && lineOrderId.intValue() == orderId) {
                int seq = ((Number) l.get("sequence_num")).intValue();
                if (seq > currentSeq && seq < nextSeq) {
                    nextSeq = seq;
                    nextLineId = ((Number) l.get("line_id")).intValue(); // Found the next station!
                }
            }
        }
        
        // Step 3: Execute the move
        if (nextLineId != -1) {
            assignOrderToLine(orderId, nextLineId); // Updates order in DB and logs a movement history
        } else {
            // No lines left? The order is finished!
            mfg.update("production_orders", "production_order_id", orderId, Map.of(
                "order_status", "Completed",
                "current_line_id", -1 // Clear the line
            ), "mfg_lead");
        }
    }
}
```

## 6. Summary of Design Principles Enforced

1. **Model-View-Controller (MVC):** Global architecture separating `panels/` (Views) from `service/` (Controllers).
2. **Composite Pattern:** Used in `JSONUtil` to represent infinitely deep BOM trees.
3. **Singleton Pattern:** Used in `BOMService` and `InventoryApiServer` to guarantee unique state control.
4. **Facade Pattern:** `BOMService` wraps all raw Map/List DB abstractions into simple Java methods for the UI.
5. **Observer Pattern:** UI components (like `JTable` listeners) react dynamically when underlying states/selections change.
6. **Data Transfer Object (DTO):** Using inner classes (like `OrderItem`) to pipe clean data to UI Comboboxes.
7. **Proxy Pattern:** `InventoryApiClient` acts as a network proxy so UI buttons don't have to write HttpURLConnection logic.
8. **Single Responsibility Principle (SOLID):** Views only paint. Utilities only parse. Services only calculate.
9. **Dependency Inversion Principle (SOLID):** Java code strictly relies on the abstract `Manufacturing` Subsystem SDK rather than binding to native SQL drivers.
